# DeepSearch

`/deepsearch` 요청을 다단계 병렬 조사로 수행한다. 각 패스에서 서로 다른 관점을 넓게 수집하고, 다음 패스에서 부족한 영역만 심화한다. 모든 핵심 주장에는 출처 링크를 붙이고, 최신성에 민감한 정보에는 정확한 날짜를 명시한다.

이 스킬 안에서는 다른 스킬을 호출하거나 참고하지 않는다. 별도 `references/` 파일도 전제하지 않는다.

## 핵심 규칙

- 모든 조사와 최종 답변은 출처 기반으로만 작성한다. 기억, 추측, 무출처 요약을 금지한다.
- 사용자가 `/deepsearch`를 명시적으로 쓰지 않았다면 이 스킬을 사용하지 않는다.
- 모든 출력은 한글로 작성하고, 기술 용어와 고유 명칭은 원어를 유지한다.
- 최신성에 민감한 정보는 반드시 웹에서 다시 확인하고, 상대 날짜 대신 절대 날짜를 쓴다.
- 서브에이전트는 조사 전용으로만 사용한다. 파일 생성, 수정, 삭제, 커밋, 코드 작성은 시키지 않는다.
- 서브에이전트는 현재 세션에서 사용 가능한 최고급 모델과 최고 reasoning effort로 생성한다. 도구 목록에 `gpt-5.4`가 있으면 `model="gpt-5.4"`와 `reasoning_effort="xhigh"`를 우선 사용한다.
- 가능한 한 각 패스의 서브에이전트는 한 번에 병렬 생성한다.

## 입력 파싱

사용자 메시지에서 `/deepsearch` 접두어를 제거한 뒤 다음 세 요소를 파싱한다.

1. 검색 주제: `숫자병렬`, `숫자패스` 패턴을 제외한 나머지 텍스트
2. 병렬 수 `P`: `숫자병렬` 패턴의 숫자. 없으면 기본값 `3`
3. 패스 수 `N`: `숫자패스` 패턴의 숫자. 없으면 기본값 `2`

예시:

- `/deepsearch sqlite 대안` -> 주제 `sqlite 대안`, `P=3`, `N=2`
- `/deepsearch AI 프레임워크 비교 10병렬 3패스` -> 주제 `AI 프레임워크 비교`, `P=10`, `N=3`
- `/deepsearch rust vs go 3병렬` -> 주제 `rust vs go`, `P=3`, `N=2`

주제가 비어 있으면 파일 작업 없이 한 줄로 주제를 다시 물어보고 멈춘다.

## 시작 알림

파싱이 끝나면 즉시 짧게 확인한다.

```text
[DeepSearch] 주제: {주제} | {P}병렬 x {N}패스
패스 1/{N} 시작 준비 중...
```

## 실행 흐름

### 1. 패스 1 설계

주제를 `P`개의 서로 다른 조사 관점으로 분해한다. 관점은 최대한 겹치지 않게 구성한다.

관점 설계 기준:

- 핵심 후보군 또는 대안군
- 공식 문서와 1차 출처
- 벤치마크, 성능, 가격, 라이선스 같은 정량 정보
- 도입 사례, 마이그레이션 사례, 실사용 리스크
- 최근 변경 사항, 릴리스, 정책 변화

각 관점은 "무엇을 조사할지"와 "왜 독립 관점인지"가 분명해야 한다.

### 2. 패스별 서브에이전트 실행

각 패스에서는 `P`개의 서브에이전트를 동시에 띄운다.

- `spawn_agent`를 사용한다.
- `fork_context`는 기본적으로 `false`로 둔다.
- 각 에이전트에는 그 패스에 필요한 최소한의 컨텍스트만 전달한다.
- 각 에이전트는 서로 다른 관점이나 빈 영역을 맡는다.
- 가능하면 하나의 `multi_tool_use.parallel` 호출 안에서 `P`개의 `functions.spawn_agent`를 동시에 호출한다.

서브에이전트 프롬프트에는 반드시 다음 요구를 포함한다.

- 웹 조사만 수행할 것
- 파일 수정이나 코드 변경을 하지 말 것
- 1차 출처와 신뢰할 수 있는 출처를 우선할 것
- 핵심 사실, 수치, 정확한 날짜, 출처 URL을 함께 반환할 것
- 불확실하거나 상충하는 주장, 추가 확인이 필요한 항목을 분리해서 적을 것

서브에이전트 프롬프트 예시:

```text
웹 조사만 수행하세요. 파일 수정, 코드 작성, 커밋은 하지 마세요.

주제: {주제}
현재 패스: {현재패스}/{전체패스}
담당 관점: {관점설명}

반드시 수행:
1. 공식 문서, 릴리스 노트, 표준 문서, 원저작자 글, 신뢰할 수 있는 1차/준1차 출처를 우선 조사
2. 최근 정보라면 정확한 발행일 또는 업데이트 날짜 확인
3. 핵심 사실을 bullet로 정리
4. 각 bullet마다 출처 제목과 URL 포함
5. 수치가 있으면 값과 기준 시점 명시
6. 상충하거나 불확실한 내용은 별도 섹션으로 분리

반환 형식:
- Findings
- Open questions / Conflicts
- Sources
```

### 3. 진행 추적

서브에이전트를 띄운 뒤에는 `wait_agent`로 완료를 기다린다.

- 미완료 에이전트 ID 집합을 유지한다.
- 완료 알림이 오면 사용자에게 짧게 진행 상황을 알린다.
- 각 패스가 끝날 때까지 반복한다.

진행 알림 형식:

```text
패스 {현재패스}/{전체패스}: {완료수}/{P} 완료
```

### 4. 패스 2 이상 설계

이전 패스의 결과가 모두 모이면 먼저 통합 분석한다.

다음을 식별한다.

- 언급만 되었고 깊이가 부족한 하위 주제
- 출처끼리 충돌하는 주장
- 수치, 날짜, 사례, URL이 부족한 부분
- 새로 드러난 중요한 비교축
- 최신성 재검증이 필요한 항목

그 뒤 다음 패스의 `P`개 조사 주제를 다시 설계한다.

- 이미 충분히 커버된 축은 반복하지 않는다.
- 빈 영역과 충돌 영역을 우선 배정한다.
- 이전 패스의 확인된 사실만 다음 패스 컨텍스트에 전달한다.
- 이전 패스의 추론이나 가설은 사실처럼 전달하지 않는다.

패스 2 이후도 패스 1과 같은 방식으로 병렬 실행한다.

## 본 에이전트의 직접 검증

최종 보고서를 쓰기 전에 본 에이전트가 직접 웹을 다시 확인한다.

- 최근 변경 사항, 가격, 정책, 버전, 출시 여부처럼 변하기 쉬운 정보
- 서로 충돌하는 주장
- 숫자나 날짜가 핵심 결론에 직접 영향을 주는 주장

이 검증 단계에서는 관련 URL을 직접 열어 날짜, 수치, 문맥을 확인하고 최종 결론을 정리한다.

## 최종 보고서

모든 패스와 직접 검증이 끝나면 한글 보고서를 작성한다.

기본 구조:

```markdown
# {조사 주제}

## 핵심 요약
2~4문장으로 전체 결론

## 주요 발견
주요 카테고리별 정리

## 비교 / 세부 분석
- 비교 표가 필요하면 표로 정리
- 구체적 수치와 기준 날짜 명시
- 장점, 단점, 제약, 도입 리스크 구분

## 미확인 사항 / 해석 주의점
출처 충돌, 데이터 부족, 시점 한계

## Sources
- [출처 제목](URL)
- [출처 제목](URL)
```

사용자가 "표로", "간단히", "목록으로" 같은 형식을 요구하면 그 형식에 맞춘다. 다만 출처 링크와 핵심 날짜는 유지한다.

## 실패 처리

- 서브에이전트 하나가 실패하면 더 좁은 프롬프트로 한 번 재시도한다.
- 동일 패스 결과가 과하게 중복되면 다음 패스에서는 관점을 더 직교적으로 다시 짠다.
- 서브에이전트가 웹 조사를 못 하는 환경이면 본 에이전트가 직접 웹 검색으로 대체하되, 병렬성이 줄었다는 점만 짧게 알린다.
- 충분한 출처가 모이지 않으면 결론을 단정하지 말고 부족한 근거를 명시한다.
