# DeepSearch: 멀티패스 병렬 웹검색

여러 서브에이전트를 병렬로 투입하여 웹검색을 수행하고, 각 패스의 결과를 분석하여 다음 패스에서 빈 영역과 심화 영역을 보완하는 반복적 조사 도구이다.

서브에이전트 10개를 동시에 보내면 한 사람이 10번 검색하는 것보다 훨씬 다양한 관점의 정보를 동시에 수집할 수 있고, 여러 패스를 거치면 1패스에서 놓친 부분을 2패스에서 메우는 식으로 조사의 깊이가 점진적으로 깊어진다.

## 입력 파싱

사용자 입력: `$ARGUMENTS`

위 입력에서 다음 세 가지를 추출한다:

1. **검색 주제**: `숫자병렬`, `숫자패스` 패턴을 제외한 나머지 텍스트
2. **병렬 수(P)**: `숫자병렬` 패턴에서 숫자 추출. 없으면 기본값 **3**
3. **패스 수(N)**: `숫자패스` 패턴에서 숫자 추출. 없으면 기본값 **2**

예시:
- `sqlite 대안 찾아줘` → 주제: "sqlite 대안", P=3, N=2
- `AI 프레임워크 비교 10병렬 3패스` → 주제: "AI 프레임워크 비교", P=10, N=3
- `rust vs go 3병렬` → 주제: "rust vs go", P=3, N=2

## 실행 흐름

### 시작 알림

파싱 결과를 사용자에게 간략히 확인한다:

```
[DeepSearch] 주제: {주제} | {P}병렬 x {N}패스
패스 1/{N} 시작: 서브에이전트 {P}개 실행 중...
```

### 패스 1: 초기 탐색

주제를 **P개의 서로 다른 관점**으로 분해한다. 각 관점이 최대한 겹치지 않으면서 주제를 넓게 커버하도록 설계하는 것이 핵심이다.

관점 분해 예시 (주제: "SQLite 대안", P=3):
- 관점 1: 분석 특화 임베디드 DB (DuckDB 등)
- 관점 2: SQLite 포크/확장 프로젝트 (libSQL, Turso 등)
- 관점 3: 키-밸류 임베디드 DB (RocksDB, LMDB 등)
- 관점 4: 모바일/클라이언트 측 대안 (Realm, ObjectBox 등)
- 관점 5: 최신 벤치마크, 성능 비교, 실제 마이그레이션 사례

P개의 서브에이전트를 **반드시 하나의 메시지에서 모두 동시에** 실행한다:

```
Agent(
  description="관점에 대한 3~5단어 설명",
  prompt="웹검색만 수행하세요. 코드 작성이나 파일 수정은 하지 마세요.\n\n{해당 관점에 대한 구체적이고 상세한 검색 지시}\n\n검색 결과를 정리해주세요: 핵심 사실, 구체적 수치, 출처 URL을 반드시 포함해주세요.",
  run_in_background=true,
  model="opus"
)
```

서브에이전트 실행 시 반드시 지켜야 할 사항:
- `run_in_background=true` — 병렬 실행을 위해 필수
- `model="opus"` — 최고급 모델 사용
- 각 에이전트의 프롬프트에 "웹검색만 수행" 명시
- 하나의 메시지에 P개의 Agent 호출을 모두 포함 (순차 실행 금지)

서브에이전트 완료 알림이 도착할 때마다 사용자에게 진행 상황을 알린다:

```
패스 1: {완료수}/{P} 완료. 나머지 {남은수}개 대기 중.
```

### 패스 2~N: 심화 탐색

이전 패스의 **모든** 결과가 도착한 후에 다음 패스를 시작한다.

#### 1단계: 결과 분석

수집된 정보를 종합하고 다음을 식별한다:
- 언급만 되고 깊이가 부족한 주제
- 상충되는 정보가 있어 추가 확인이 필요한 부분
- 구체적 수치/사례/출처가 부족한 영역
- 이전 패스에서 새롭게 발견된 중요한 하위 주제

#### 2단계: 개선된 검색 쿼리 설계

식별된 빈 영역을 P개의 구체적 검색 주제로 분배한다. 이전 패스에서 이미 충분히 커버된 내용은 반복하지 않는다. 각 서브에이전트에게 이전 패스에서 발견된 관련 사실을 컨텍스트로 전달하여, 더 정확하고 심화된 검색이 가능하도록 한다.

#### 3단계: 실행

패스 1과 동일한 방식으로 P개의 서브에이전트를 동시에 실행한다.

```
패스 {현재}/{N} 시작: 서브에이전트 {P}개 실행 중...
```

이 과정을 N패스까지 반복한다.

### 최종 보고서

모든 패스 완료 후, 수집된 모든 정보를 종합하여 **한글 보고서**를 작성한다.

보고서 기본 구조:

```
# {조사 주제}

## 핵심 요약
2~3문장으로 전체 결론

## 본문
카테고리별로 구조화된 상세 내용
- 비교 표 (해당되는 경우)
- 구체적 수치/데이터
- 장단점 분석

## Sources
- [출처 제목](URL)
- ...
```

사용자가 보고서 이외의 형식을 요청한 경우 그에 맞춘다. 예를 들어 "목록으로", "표로", "간단히" 등의 요청이 있으면 해당 형식으로 출력한다.

## 규칙

- **언어**: 모든 출력은 한글. 기술 용어와 코드 식별자는 원어 유지
- **모델**: 모든 서브에이전트는 `model="opus"` 사용
- **병렬 실행**: 반드시 하나의 메시지에서 P개의 Agent 호출을 모두 포함. 순차 실행 금지
- **웹검색 전용**: 서브에이전트는 웹검색과 정보 정리만 수행. 파일 생성/수정 금지
- **팩트 기반**: 출처가 있는 정보만 보고서에 포함. 추측 금지
- **진행 알림**: 각 패스의 진행 상황을 사용자에게 간결하게 알림
- **중복 방지**: 이후 패스에서 이전 패스와 동일한 검색을 반복하지 않음
